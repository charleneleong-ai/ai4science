# The bond-valence tier was also measuring the wrong domain (2026-07-17)

After fixing the geometry prior ([writeup](2026-07-13-geometry-prior-wrong-domain.md)), I ran the
same test against **every** always-on tier — *does it accept real metalloprotein sites?* Three of the
four passed. The fourth did not:

| always-on tier | verdict on real MetalPDB sites | |
|---|---|---|
| geometry | 96–99% trust | ✅ (fixed 2026-07-13) |
| coord_symmetry (nVECSUM) | 100% trust | ✅ |
| coord_geometry (polyhedron RMSD) | 81–99% trust | ✅ |
| **bond_valence** | **defers 64% of real Ni²⁺ sites** | ❌ |

Same bug as geometry, in a second always-on tier — and always-on means **any defer collapses the
whole consensus**, so this was quietly tanking most real Ni designs regardless of the other tiers.

## Why

The R₀ parameters were **Brese & O'Keeffe 1991 — small-molecule crystals**. There, Ni–O bonds are
~2.05 Å; the R₀ is tuned so a site at that length sums to the formal valence. Real protein Ni sites
average 2.19 Å and are lower-CN (entatic strain is the norm), so the **median real Ni²⁺ site sums to
BVS ≈ 1.03 against a formal valence of 2** — under-bonded by nearly a full valence unit, purely from
the domain mismatch. The shipped params file's own source note had already flagged it: *"verify
against bvparm2020.cif before quantitative use."*

## The fix — and it took two corrections, not one

**1. R₀ (the bias).** A per-metal additive shift, chosen so the *median* real-site BVS lands on the
formal valence. Median, not least-squares: a per-element fit overfit the scatter and produced
nonsense on sparse donors (Ni–S, 74 bonds → a *negative* shift). The uniform shift preserves the
literature's physical N/O/S offsets and corrects only the domain-wide bias. Ni: R₀ +0.245 Å.

**2. Tolerances (the scatter).** Fixing the bias alone still deferred ~42% of real Ni sites, because
protein BVS is *intrinsically scattered* — the sum conflates coordination number with bond length,
and protein coordination is heterogeneous (CN 3–7, variable lengths). So the tolerances
(`trust_tol`, `ood_tol`), set for tight small-molecule BVS, were *also* wrong-domain. They now come
from the **real-site |BVS − valence| distribution** (the data owns the tolerance, as `cn_range` does
for geometry), and are **per-metal** — Cu's BVS is tighter than Ni's or Co's.

Real-site defer rate, per metal: **64% → 10% (Ni), 30% → 10% (Cu), 39% → 9% (Co).**

## The honest coda: BVS is a weak filter in this domain

This is the same lesson geometry taught, and it has to be said plainly. To stop rejecting real Ni/Co
sites, the tolerances had to widen so far that BVS **cannot** finely discriminate: a moderately
under-bonded Ni site (every bond +0.5 Å) is *deferred* only ~28% of the time (though ~90% drop below
`trust` to `weak`). Over-bonding stays crisp for every metal; under-bonding *deferral* is
metal-dependent and weak for Ni/Co — it downweights, but rarely collapses the consensus on, a
moderately under-bonded site.

That is not a failure of the fix — it is the true resolution of the tier. BVS in the protein domain
catches **egregious** over/under-bonding, not fine distinctions. The discrimination comes from the
ensemble (geometry + coordination + precedent + MLIP), not from any single tier. The small-molecule
params only *looked* sharp because they were scoring protein sites against the wrong reference —
exactly as the CSD geometry prior did.

## What changed

- [`scripts/build_bond_valence_params.py`](../../scripts/build_bond_valence_params.py) — derives
  protein-domain R₀ + per-metal tolerances from MetalPDB (solvent-excluded, ≥30-site floor, refuses
  to write an empty table — the same guards as the geometry builder).
- [`bond_valence_params.json`](../../src/touchstone/data/bond_valence_params.json) — recalibrated, with
  per-metal tolerances and a `domain` tag on each metal.
- [`BondValenceVerifier`](../../src/touchstone/geometry/bond_valence.py) — reads per-metal tolerances
  from the params (constructor args are now only a fallback).
- Calibration is a test ([`TestBvsCalibration`](../../tests/test_bond_valence.py)): real sites defer
  ≤15%, and the old small-molecule params are asserted to fail it — 360 real MetalPDB sites,
  regenerated by [`build_bvs_fixture.py`](../../scripts/build_bvs_fixture.py).
- The `ideal_nickel_site.pdb` benchmark fixture moved from the small-molecule 2.09 Å to the
  protein-domain 2.30 Å (where a CN6 Ni–N₃O₃ site sums to valence 2 *and* geometry trusts).

## The pattern (fifth occurrence, and the audit that found it)

MetalHawk, MACE/OrbMol, the geometry prior, the precedent table — and now the BVS params. Every time,
a component validated in one domain and used in another. The new thing here is **method**: after
fixing geometry, I ran the *same* real-site test against every remaining always-on tier, and it found
this one directly rather than by accident. The check — *does this tier accept ground truth it must not
reject?* — is now cheap to apply to any tier we add.
